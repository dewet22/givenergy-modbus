    v2.1 roadmap

    v2.1 release roadmap¶

    A working document. Expected to change as alphas land and consumers (givenergy-hass, givenergy-cli) report back. The intent is a chain of small, frequent prereleases rather than a single "land everything, then release" push — each alpha self-contained enough that scope can move between alphas, or fall out to v2.2, without blocking the line.

    Reconciled 2026-05-31 against shipped reality (a3 → a11). The original a4/a5/b1 theme plan was overtaken by a real-user-driven arc (EMS support for a live install via givenergy-hass#52), which produced more than the plan pencilled in — including the wire-capture fixtures and golden-master harness originally slated for beta. The cadence philosophy held; the specific themes didn't. "What's next" below is now re-derived from the actual open issues rather than the superseded plan.

    Cadence philosophy¶

    • Release often, iterate visibly. The PyPI alpha publishes downstream dispatch events to givenergy-hass and givenergy-cli, so each cut becomes a tester checkpoint. This worked exactly as intended — Nick's live EMS install drove a4–a11 through fast feedback on each alpha.
    • Self-contained alphas. If scope slips on one item, it falls out and becomes a v2.2.0 candidate rather than gating the line. Better a shipped narrower release than a wider one that never ships.
    • Investigation items ride alongside but don't gate any release. They produce findings and shape later design decisions.
    • Documentation: each alpha's changelog section is generated from conventional-commit prefixes; the release workflow walks the range since the previous tag (see scripts/release.py).

    Shipped (a3 → a11)¶

    What actually landed. Grouped by theme rather than strict per-alpha, since the work cut across alphas.

    EMS support — the dominant arc, driven by a live EMS + 2-inverter install: - EMS-aware polling: skip inverter-style banks on EMS controllers (#93, a4); read + sanity-check the IR(2040) rollup at detect (#109/#111, a5). - Device addressing redesign: 0x11 is the inverter's canonical address, not 0x32; removed the blanket rewrite, made addressing model-derived (#119/#121, a5). Live-validated on HYBRID_GEN1 hardware (reads 0x31, writes 0x11). - EMS plant writes: charge/discharge/export slots, SoC targets, export power limit (#134, a7); per-endpoint + EMS-named export setters for HA entity parity (#138 a8, #143 a10); plant_enabled read-back (#145, a11). - EMS serial-format recognition to silence per-poll warnings (a7).

    Wire-capture fixtures + golden master (was planned for b1; landed early): - Real redacted EMS/hybrid/AIO captures with per-plant READMEs (#103 a4, #120 a5). - Date-preserving serial redaction (#113/#116, a5). - Golden-master regression tests replaying fixtures through the full classification/decode machinery (#137, a7); register-node documentation (#140, a9).

    Resilience + correctness: - RefreshPartiallySucceeded / RefreshFailed partial-failure contract (#125, a5). - Contended-bus refresh defaults 2.0s/1 retry (#132, a9). - Per-attribute display precision (#129, a6); unified Inverter facade for EMS-managed topology (#98, a4).

    Infra: lean CI matrix (Linux per-PR, cross-OS weekly, a5); uv security bump (a9); release-workflow hardening (a3).

    What's next¶

    Re-derived from the open issue tracker. No fixed alpha assignments — pick by what a consumer needs next or what unblocks the beta.

    Original a4/a5 items never actioned (still open, still wanted): - #74 — re-apply reverted v2.0 model-layer correctness fixes (needs the history-dig first). - #76 — unify the battery-energy registers behind computed properties. - #65 — RegisterCache freshness timestamps (record-only; invalidation policy stays a v2.2 item). - #75 — re-scoped: the command mixin landed; what remains is the per-model write-safety contract (reject setters that don't apply to the detected model).

    Surfaced by the EMS arc: - #144 — identify registers for self-heating / winter conditioning / EPS-enable / export-priority (blocked on GE-portal write-captures; the rest of the EMS-parity request shipped). - #117 — cross-frame redactor serial-leak edge (a genuine redaction gap worth closing). - #108 — EMS rollup bitfield decode mismatch against real capture. - #48 — field-validate register mappings inherited from britkat's fork (open-ended audit).

    Path to 2.1.0b1¶

    The line has had 11 alphas and the model layer has settled — the original beta criterion ("API stable, done rearranging the model layer") is largely met. Before promoting to beta:

    • Confirm the EMS write path on live hardware. #134 shipped EMS writes correct-by-construction + register-safety-reviewed, but they haven't been exercised against a real EMS. The live write-test is being driven from givenergy-hass#52; a clean result is the last gate on the EMS feature.
    • Decide #106 (graph-shaped Plant) in-or-out. It's the one large architectural item left. If it's a v2.2 effort (likely — it's a big abstraction shift), say so explicitly so it stops shadowing the beta.
    • Pick up whichever of the "what's next" items are tractable; let the rest fall to v2.2 rather than gate the beta.

    Beta then promotes the line out of "still rearranging the model layer" into "API stable, fix bugs and tighten contracts", with public-API compatibility tests and register provenance/confidence (post-v2 §5/§10) as the beta's own scope.

    2.1.0rc1 → 2.1.0¶

    Soak period. No new scope; only bug fixes from beta-tester feedback. Final cut when consumers report clean.

    Investigation threads (no release gating)¶

    • #91 — detect frozen BMS cache during firmware updates (needs firmware-update wire captures).
    • #147 — BMS block dropouts on the battery sub-bus (Pattern B from #78; needs paired RS485 capture).
    • #114 — undecoded function #57 frame; #100 — WO-prefix LAN-config broadcast; #115 — function #134 Gen1 BPM workaround.
    • #141 — three-phase grid-power register nodes (needs a 3-phase capture).
    • Internal refresh serialization (post-v2 §3) — needs the locking-model design first; likely a v2.2 item.

    Out of scope for v2.1¶

    If something falls out and won't fit before beta, it becomes a v2.2 candidate. Better a shipped narrower v2.1 than a perpetually in-progress one. Likely v2.2:

    • The graph-shaped Plant abstraction (#106) if it doesn't make the beta cut. Design and phased rollout: Graph-shaped Plant (Phase 1 — the typed-device enumeration API — has landed).
    • Freshness-aware RegisterCache invalidation policy (the layer above #65's recording).
    • Internal refresh serialization (post-v2 §3) if the design hasn't matured.
