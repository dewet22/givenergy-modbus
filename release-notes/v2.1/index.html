    V2.1

    givenergy-modbus v2.1 — ecosystem release notes¶

    v2.0 was the reunion — pulling the fork's protocol and device-model work back into a maintained line. v2.1 is what happened once real plants started feeding data back in. Almost everything here was driven by an actual reported topology rather than a planning doc: an EMS-managed multi-inverter site that the flat model couldn't represent, an All-in-One that was being mis-detected as three-phase, a CRC that had never been validated on the response side. The release is less "new features" and more "the library now matches what the hardware actually does, with the evidence to back it".

    TL;DR — what actually changed for my plant?¶

    If you have an EMS-managed plant (a GivEnergy EMS controller coordinating two or more inverters — common in larger or commercial installations), this is the first release where that topology works properly. The library can now detect it, poll it without timing out, and write charge / discharge / export schedules to it.

    If you have a residential All-in-One, it was being misidentified as a three-phase unit, which caused most readings (SoC, voltages, firmware version, charge slots) to show as zero or null. That's fixed.

    Energy readings are more accurate . Two values that were historically mislabeled have been corrected: "load today" (IR35) was actually AC-charge-today, and "inverter output today" (IR44) was actually PV-generation-today. Both are renamed in the library. A new "consumption today" value is now derived the same way the GivEnergy app calculates it — useful if your integration was computing it manually or relying on the wrong register.

    More controls are accessible. Smart-load scheduling slots, inverter parallel mode, and a handful of other registers that weren't in the safe-write list now are, confirmed against the GivEnergy app's own register map.

    The communication layer also got some quiet fixes — response checksums are now validated end-to-end, and the All-in-One protocol quirk that was causing it to ignore requests entirely is resolved. These won't change any numbers you see, but they reduce the chance of silent data corruption passing unnoticed.

    The related and dependent tools in this ecosystem have been updated as well:

    givenergy-hass 1.1, the native Home Assistant integration¶

    v1.1 is the release where the integration grows from a solid hybrid-inverter tool into one that covers most of the GivEnergy fleet. Riding the move onto the givenergy-modbus 2.1 library, it brings first-class support for EMS/Flexi plants — the controller now gets its own device and a tailored dashboard, with charge, discharge and export slot scheduling and per-slot SoC targets exposed as proper entities — along with AC-coupled inverters and All-in-One units (export priority, EPS, and AC battery limits, correctly gated to the hardware that actually has them), and Smart Load slot scheduling.

    Just as importantly, it's the release where imperfect hardware stops toppling the integration: a partial poll now loads what it can and marks only the failed reads unavailable rather than failing the whole setup and looping; a slow-responding battery no longer drops out of the detected topology; and the plant's shape persists across restarts. Rounding it out are sensors rendered at their true register precision, a cross-pack Battery Health dashboard with a bundled cell-voltage heatmap, and a documented path — catalogue plus script — for carrying your long-term statistics across from GivTCP. Upgrading from 1.0.x takes nothing by hand; entities and devices appear on the first poll after the update.

    givenergy-cli 1.4, the command-line Swiss army knife¶

    The CLI grows from a three-command viewer into a proper debugging toolkit. The headline additions are probe, for issuing raw register reads against arbitrary register blocks and devices (handy for poking at undocumented blocks, and it takes hex or decimal addresses interchangeably), and mock-server, which replays a previously-recorded wire capture as a faithful in-memory inverter so you can drive other clients without needing the actual hardware — fidelity I've validated by pointing the official GivEnergy Android app at it.

    Under the hood, the givenergy-modbus 2.1 line brings a more honest failure model: a partly-failed poll reports exactly which reads dropped rather than collapsing into a generic timeout. Rounding it out are the usual housekeeping touches — proper PyPI project links and a reworked README documenting the full command set.

    What's new, in more detail¶

    EMS plant controller support¶

    The headline. The flat "one inverter per connection" model couldn't represent an EMS-managed plant (a controller coordinating multiple inverters, meters and batteries), so a whole strand of v2.1 went into making EMS a first-class topology:

    • EMS-aware polling — the controller doesn't expose the inverter-style register banks, so detection skips them instead of timing out every cycle, and reads the EMS rollup at IR(2040) to sanity-check the plant.
    • EMS plant-level writes — charge / discharge / export slots, per-slot SoC targets, and the export power limit, via dedicated set_ems_* commands.
    • A unified Inverter facade so an EMS-managed plant still resolves a sensible inverter view for consumers built around the single-inverter shape.

    A caveat I'd rather state plainly: the EMS write path has had a register-safety review, round-trips through the mock, was exercised against the GivEnergy app's own protocol, and matches the registers the cloud dongle writes in captured traffic — but it has not yet been confirmed against live EMS hardware by writing through this library. The write path is the same WriteHoldingRegister at 0x11 that the inverter writes already use on real hardware, so I'm reasonably confident, but "reasonably confident" is not "verified".

    Device addressing, model-derived¶

    0x11 is the inverter's canonical address, not the 0x32 the older code assumed. The blanket address rewrite is gone; addressing is now derived from the detected model (0x31 for AC / HYBRID_GEN1, 0x11 elsewhere). This was live-validated on HYBRID_GEN1 hardware before it shipped.

    A real partial-failure contract¶

    refresh() and load_config() now raise RefreshPartiallySucceeded / RefreshFailed instead of silently returning a half-populated plant. refresh_plant() and watch_plant() are deprecated in favour of calling detect() once and then driving your own loop over load_config() / refresh() — which is what the deprecation warning points you at. They'll be removed in 3.0.

    CRC integrity, end to end¶

    Two pre-existing defects got fixed. The request CRC wasn't including the device-address byte (surfaced while debugging an All-in-One that wouldn't answer at all). The response CRC was never validated — the check was a stub. Both now run through one shared _update_check_code over the same byte span, so request and response can't drift apart again. The redactor that protects captured wire data was reworked to be frame-aware (decode → redact by field → re-encode with a correct CRC) rather than a byte-level regex, and the undocumented plaintext LAN-config broadcast the dongle emits every few minutes is now decoded as a first-class frame so it redacts properly too.

    Testing infrastructure that doubles as a discovery tool¶

    • givenergy_modbus.testing.MockPlant — a faithful in-memory mock that seeds per-device register state from a committed capture and synthesises correct-CRC responses to any read. It's good enough that the real GivEnergy Android app connects to it and believes it's talking to an inverter, which is what makes the next item possible.
    • Register cross-correlation helpers (sentinel_devices, identify) — with GivEnergy gone, there's no register documentation. So instead: seed the mock with sentinel values, point the app at it, read the displayed values off the app's screens, and invert the conversion to recover which register backs each label. The method and a worked procedure are in docs/register-identification.md. This is how the writable register surface below was confirmed.

    Register map, corrected against the app¶

    The GivEnergy app's "Direct Control" screen turned out to enumerate every user-writable register with its name — effectively the documentation that no longer exists elsewhere. Cross-referenced against it (and against GivTCP for corroboration), v2.1:

    • expands WRITE_SAFE_REGISTERS with the app-confirmed writable set;
    • renames HR(199) from enable_standard_self_consumption_logic (an inherited guess that GivTCP shared) to enable_inverter_parallel_mode, the app's actual name, keeping the old name working as a deprecated alias;
    • adds the Smart Load scheduling slots (smart_load_slot_1..10, HR 554–573) with decode and write helpers.

    All-in-One single-phase fix¶

    A residential All-in-One was being treated as three-phase: polled for a per-phase register bank it doesn't have, and decoded with a layout that zeroed ~30 of its real fields. It's HV but single-phase; removing it from the three-phase sets fixed both the polling and the decode. Confirmed against a real AIO packet capture and the owner.

    2.1.1 patch¶

    Two decode corrections, both confirmed via the cross-correlation tooling above:

    EMS per-slot inverter status (#108). The per-slot inverter status field in the EMS rollup (IR(2045), 4×3-bit) was passing raw codes through the inverter Status enum, whose values don't match this field's encoding — a present, idle inverter (code 2) was reading "WARNING". The fix surfaces the raw 3-bit code as a hex string (the house idiom for uninterpreted values; see device_type_code, fault_code) and adds a inverter_N_suspected_status field with the two verified interpretations: 0=ABSENT, 2=ONLINE. Use inverter_count for reliable presence; the suspected-status field is advisory.

    e_load_day mislabel (#174). IR(35) was documented in GivTCP as house-load-today but sentinel cross-correlation proved it is AC-charge-today. Renamed to e_ac_charge_today with no deprecated alias (the old name was actively wrong). Similarly, IR(44) (e_inverter_out_day) is PV-generation-today, renamed to e_pv_generation_today with a deprecated @property alias for one release. A computed e_consumption_today field is added to SinglePhaseInverter using the GE-app formula:

    e_consumption_today = e_pv_generation_today + e_grid_in_day − e_grid_out_day − e_ac_charge_today

    This matches what the app calls "Consumption today": consumption is derived, not metered, on single-phase units. Three-phase has a native e_load_today register and is unaffected.

    Migrating from v2.0¶

    • HR(199) field rename. enable_standard_self_consumption_logic → enable_inverter_parallel_mode. The old name still resolves via a deprecated property (with a DeprecationWarning) but no longer appears in model_dump(). If you build entities by iterating the dump, switch to the new name; if you read the attribute directly, it keeps working for now.
    • IR(35) field rename (2.1.1). e_load_day is gone — no alias, it was actively wrong. Switch to e_ac_charge_today. Any code that iterated model_dump() and expected e_load_day will simply no longer see it.
    • IR(44) field rename (2.1.1). e_inverter_out_day → e_pv_generation_today. The old attribute still resolves with a DeprecationWarning for one release (both inverter classes), pointing at e_pv_today on three-phase units.
    • New field: e_consumption_today (2.1.1). Appears in model_dump() on SinglePhaseInverter only. Returns None when any input register is absent.
    • Refresh API. refresh_plant() / watch_plant() are deprecated; move to detect() + your own load_config() / refresh() loop, handling RefreshPartiallySucceeded / RefreshFailed.
    • Additive, no action needed: the smart_load_slot_* fields and the expanded writable register set.

    What's coming next¶

    v2.2 is open as the active line. The big one is #106 — a graph-shaped, typed-device Plant model (inverter / EMS / gateway / battery / meter as first-class devices rather than everything resolved through the inverter lens). It's the architectural shift that the EMS, gateway and three-phase symptoms all point at, and it's large enough that it gates a hass entity-naming migration, so it's the v2.2 flagship rather than something squeezed into a patch. Alongside it: the per-model write-safety enforcement (#75), and continued register-map verification (#48) now that there's a tool for it.
